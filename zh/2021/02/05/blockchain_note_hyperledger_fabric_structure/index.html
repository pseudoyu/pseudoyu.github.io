Hyperledger Fabric详解

作者： Arthur / 2021-02-05 / 分类： Programming

blockchain, programming

翻译： EN

Hyperledger

Hyperledger是一组专注于跨行业分布式技术的开源项目

  • 创建企业级、开源、分布式的分类框架和代码库来支持业务用例
  • 提供中立、开放和社区驱动的基础设施
  • 建立技术社区并推广，开发区块链和共享账本概念验证、使用案例、试验和部署
  • 建立行业标准，鼓励更多企业参与到分布式账本技术的建设和应用中来，形成一个开放的生态体系
  • 教育公众关于区块链科技的市场机会

设计理念

  • 对加密货币的作用存疑
    • Hyperledger项目是独立的，不使用加密货币、山寨币等代币
  • 配备完整的API
    • 所有Hyperledger项目都提供丰富且易于使用的API，支持与其他系统的交互
  • 模块化
    • Hyperledger正在开发模块化的、可扩展的框架（可重用的通用构件）
  • 高度安全
    • 安全性是分布式账本技术的一个关键考虑因素，尤其是许多应用场景涉及高价值交易或敏感数据
    • Hyperledger项目在系统设计上就充分考虑安全性，支持企业级应用场景
  • 交互性
    • 未来很多不同的区块链网络将会有沟通和交换数据的需求，以形成更复杂和强大的网络
    • 因此，在Hyperledger项目中，大多数智能合约/应用需要具备跨区块链网络的可移植性

框架

  • Burrow
  • Fabric
  • Grid
  • Indy
  • Iroha
  • Sawtooth

工具

  • Hyperledger Cello
    • 方便地创建、管理区块链服务
    • 降低超级账本框架部署、维护的复杂度
    • 用来搭建区块链BaaS平台
    • 允许将区块链部署到云端
    • 可以通过仪表盘来创建和管理区块链，技术人员也可以方便地创造区块链实例
    • 将SaaS部署模型引入到区块链生态系统，帮助企业进一步开发和部署超级账本框架
  • Hyperledger Explorer
    • 可视化区块链操作工具
    • 首个超级账本区块链浏览器
    • 旨在创建用户友好的Web应用程序
    • 用户可以查看、调用、部署或查询区块、交易和交易相关数据、网络信息、智能合约及存储在账本中的其他相关信息

Hyperledger Fabric

  • Fabric是一个模块化、可扩展的开源区块链系统，用于部署和操作联盟链（有权限的）
  • 支持模块化的共识协议
  • 由Linux基金会维护
  • 第一个采用标准通用编程语言（Java/Go）编写分布式应用的系统，并且没有对加密货币的系统依赖性
  • 它提供了一种解决方案来保护具有共同目标但彼此不完全信任的一组实体之间的交互，例如：交换资金、商品或信息的企业

架构

大部分区块链架构为Order - Execute - Validate - Update State，如比特币区块链

  • 采用PoW机制对Block进行排序
  • 区块链网络中的每个节点逐个验证
  • 更新状态

Fabric采用了Execute - Order - Validate - Update State架构

  • Execute
    • 本地模拟交易并背书
    • 创建Read-Write Set
    • 收集背书信息/
  • Order
    • 排序Read-Write Sets
    • 广播（通过共识机制）
  • Validate
    • 验证背书信息和Read-Write Sets
    • 去除无效/冲突交易
  • Update State
    • 更新状态

  • Fabric区块链由节点组成网络
  • 架构是要求许可的，所有参与Fabric网络的节点都有身份（由模块化的MSP, Membership Service Provider提供）

分布式账本解决平台，采用模块化架构，支持不同组件的可插拔实现以适应业务系统的复杂性，并且通过授权方式与通道机制使得安全和隐私保护方面达到了商用化的标准。首次将权限管理机制引入区块链领域，其可拓展的架构设计、开放的接口风格、可插拔的组件化实现为分布式账本平台的设计和实现提供了重要参考。

MSP成员服务提供商

  • MSP确定可信任中间CA和根CA，用于验证受信任域的成员
  • 有一个独立的Fabric CA模块，用来管理证书服务，也允许第三方CA的接入
    • Fabric CA提供客户端与SDK两种方式与CA进行交互
    • 每个Fabric CA都有一个根CA或者中间CA，可采用集群搭建中间CA以保证CA的安全性
    • 一般采用根证书、业务证书、用户证书三级证书结构，所有的下层证书都会继承根证书的信任体系
      • 根证书用来签发业务证书
      • 业务证书用来签发具体的用户证书
        • Enrollment CA属于身份认证证书
        • Transaction CA属于交易签名证书
        • TLS CA属于安全通讯证书
  • 通过身份认证弱化了节点作恶，在性能和隐私保护上满足政府企业需求

通道Channel

  • Channel是一个独立的账本/区块链，有不同的世界状态
  • 一个Peer节点可以加入多个channel

Chaincode智能合约

  • Fabric中的智能合约称为链码（chaincode），由外部应用程序与分布式账本进行交互
  • Fabric使用Docker存放链上的代码，而不需要依靠特定的虚拟机，Docker为链码执行提供了一个安全、轻便的语言执行环境

共识机制

区块链网络中，不同参与者之间交易必须按照发生的顺序写到分布式账本中，区块链网络中，不同参与者之间交易必须按照发生的顺序写到分布式账本中

  • SOLO（只限于开发）
  • Kafka（一种消息平台）
  • Raft（相比Kafka更中心化）

P2P协议

  • Peer节点和排序节点采用gRPC对外提供远程服务供客户端进行调用
  • 网络中的节点之间通过Gossip协议来进行状态的同步和分发
    • 用于网络内多个节点之间的数据分发或信息交换
    • 容易实现、容错率高，被分布式系统广泛采用
    • 原理：数据发送方从网络中随机选取若干节点，将数据发送过去，接收方重复这一过程（除发送方外），网络中所有节点最终会达成一致（复杂度为节点总数的对数）

分布式账本

  • 世界状态：记录最新的分布式账本状态
  • 事务日志：记录世界状态的更新历史

存储数据库

  • LevelDB
  • CouchDB

应用编程接口

  • SDK 开发工具包
  • CLI 命令行

Fabric网络核心逻辑角色

  • Clients
    • 提交交易Proposal，帮助安排执行阶段，最后广播交易至排序节点
  • Peers
    • 执行交易提案，验证交易
    • 所有节点都维护区块链账本（记录所有交易、状态的数据结构，只能添加数据）
    • 只有背书节点（Endorsers）执行所有的交易Proposal，由Chaincode制定策略）
  • OSN(Ordering Service Nodes)
    • 由排序节点组成，提供排序服务
    • 为Fabric中的所有交易排序，每个交易包含状态更新、执行环节产生的依赖、背书节点的数字签名等
    • 不参与交易执行/验证

Fabric网络中的角色是逻辑角色

  • 一个Peer节点可能承担多个角色
    • 部分chaincode的背书节点
    • 排序节点
  • 一个角色可能会被多个节点承担
    • Chaincode的背书节点可能有多个（如要求3/5）
    • 可能有多个排序节点，通过一些机制共同对交易进行排序

优势

  • 将复杂环节功能分配至各个逻辑角色节点，消除了网络瓶颈，增强了拓展性
  • 可插拔的共识机制（1.4版本支持Raft共识机制）、权限管理、加解密、账本机制等模块
  • 灵活的信任模型（不同交易可以根据不同的背书策略进行验证）
  • 加强了身份证书管理服务，作为单独的Fabric CA 项目，提供更多功能
  • 支持多通道特性，不同通道之间的数据彼此隔离，提高隔离安全性
  • 使用chaincode，支持更灵活的智能合约语言和第三方
  • 可并发执行所有操作
    • 不同的背书节点模拟执行不同的交易Proposal
    • 节点只需要验证数字签名
  • 背书节点支持非确定性的执行（如随机操作）
  • Chaincode的可信度
    • Chaincode仅在特定的背书节点上部署和执行
    • 可以隐藏商业逻辑

Fabric应用开发及交互

Fabric交易流

  1. 由Client提交交易Proposal至背书节点（特定）
  2. 背书节点模拟执行，进行背书并返回信息（Read-Write Set、签名等）
  3. Client（SDK）收集到足够的背书后（策略由Chaincode制定，如图中为两个背书）提交已背书交易（Proposal+Response)至排序节点（OSN）
  4. 排序节点将交易打包成blocks，排序（不执行/验证）并广播至所有节点
  5. 所有节点对新blocks进行验证并提交至账本

Execute执行环节

Clients（SDK）提交交易proposal后，背书节点用本地状态模拟执行，返回签名和Read-Write Set回SDK

  • Key
  • Version
  • Value

注：Fabric支持多种背书策略，SDK在提交前会验证是否满足背书要求；如果Chaincode只做了查询账本操作，Client不会提交至OSN

Transaction proposal包含

  • Chaincode智能合约
  • Chaincode输入
  • 客户端签名

Proposal response包含

  • 返回值
  • 模拟执行结果的Read-Write Set
  • 背书节点的签名

Endorsed Transaction = Transaction proposal + Proposal response

Endorser背书节点详解

  • 不同的chaincode可以有不同的背书节点（或多个）
  • 节点根据背书节点来对交易进行验证
    • 指定背书节点
    • 3/5
    • ((P1 and P2) or P3)
    • …
  • 交易验证依赖于对背书节点的信任
  • 背书节点只根据当前状态（本地区块链）检查chaincode执行是否正确
  • 背书节点只根据当前账本（本地）状态模拟执行交易并返回
  • Read-Set包含交易执行中读取的所有变量和其version
  • Write-Set包含所有被编辑的变量及其新值

Chaincode执行示例

func (t *SimpleChaincode) InitLedger(ctx contractapi.TransactionContextInterface) error {
	var product = Product { Name: "Test Product", Description: "Just a test product to make sure chaincode is running", CreatedBy: "admin", ProductId: "1" }

	productAsBytes, err := json.Marshal(product)

	err = ctx.GetStub().PutState("1", productAsBytes)

	if err != nil {
		return err
	}
}

在这个简单的Chaincode示例中，更新了key-value值

  • key: 1
  • value: Product { Name: “Test Product”, Description: “Just a test product to make sure chaincode is running”, CreatedBy: “admin”, ProductId: “1” } 的Json形式
  • Version版本经过Write操作会+1

Order排序环节

Clients提交经过背书的交易至排序节点（排序节点可通过一些共识策略组成OSN），排序节点对所有交易进行排序（不对交易内容进行核查/执行/验证），并发送至所有节点（实际运行中，交易会被打包成区块）

Orderer排序节点详解

  • 排序节点不对已背书交易进行验证
  • 仅将所有交易打包成块并排序
  • Fabric区块链网络的完整性依赖于排序节点的一致性

Validate验证环节

当节点接收到一个新区块，会对区块中的所有交易进行验证并标记

  • 是否满足背书策略
  • 是否有状态冲突，如Read-Set中的Version

Peer节点详解

  • Peer节点可能不包含chaincode
  • 交易执行的正确性依赖于对背书节点的信任
  • Peer节点对交易进行验证
    • 是否满足背书策略（指定节点/数量/…）
    • 是否有Read-Write冲突（与节点的本地账本备份对比）

Update State环节

当新区块被验证为可信，则根据Write-Set更新世界状态（每次更新会变更version number）
